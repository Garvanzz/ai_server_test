# 工会模块 DB 设计分析：MySQL vs Redis 与业界实践

## 一、当前设计概览

| 数据类型 | 当前存储 | 用途 |
|---------|----------|------|
| 公会主数据 (Guild + MemberData + Yuanchi) | **MySQL** `guild` 表 | 持久化、启动全量加载到内存；创建时按 `guild_name` 唯一校验 |
| 公会日志 (GuildLog) | **MySQL** `guild_log` 表 | 按 `guild_id` 查询最近 N 条 |
| Manager 状态 (guildList、GuildApplication 等) | **Redis** `GuildManager` 单 key | 启动时恢复内存索引（与 MySQL 公会数据一起用） |
| 玩家-公会关系 (PlayerGuild) | **Redis** `player_guild:{playerId}` | 玩家所属公会、签到/祈福/地图等状态 |
| 公会排行榜 | **Redis** sorted set | 战力等排行 |
| 公会聊天记录 | **Redis** list | 最近 N 条聊天 |

运行态：**内存为主**（所有公会 entity 在内存），MySQL 为 30 分钟分批写回；玩家公会数据通过 WriteBackCache 写回 Redis。

---

## 二、你纠结的点：MySQL 还是 Redis？

核心矛盾可以归纳为：

- **用 MySQL**：想利用「结构化、可 SQL 查询、事务、持久化可靠」。
- **用 Redis**：想利用「高性能、低延迟、适合缓存和会话型数据」。

这两种诉求都合理，但**单一选型**往往无法同时满足「复杂查询 + 高并发读写 + 持久化 + 可运维」。业界常见做法是：**按数据特征拆分职责，MySQL 与 Redis 分工协作**，而不是二选一。

---

## 三、业界常见做法（游戏/社交场景）

### 1. 持久化与一致性：用 MySQL（或 PG）做 Source of Truth

- **公会主数据**（名称、等级、配置、成员列表、元池等）需要**长期留存、可恢复、可审计**，适合放在关系型库。
- 一旦只用 Redis，会面临：RDB/AOF 的持久化策略、集群下的一致性、数据恢复与备份策略等，运维和可靠性都不如 MySQL 成熟。
- 因此常见做法是：**公会主数据在 MySQL 做主库**，Redis 只做缓存或会话层。

### 2. 高并发与热点：用 Redis 做缓存 / 会话层

- **读多写少**：公会信息、成员列表等，用 Redis 做缓存，减轻 MySQL 压力。
- **会话型 / 状态型**：玩家当前所属公会、今日是否签到、祈福状态等，以「玩家维度」访问，适合 Redis（key 设计为 `player_guild:{playerId}` 这类）。
- **排行榜、聊天**：天然适合 Redis（sorted set、list），和你们当前一致。

### 3. 结构化查询：谁来做？

- **需要复杂查询的场景**（例如：按名字模糊搜公会、按等级/人数分页、按创建时间排序、跨公会统计）：
  - 若全部放 Redis，要么用 Redis Search 等扩展，要么在应用层维护多份结构，复杂度和一致性成本都高。
  - 这类需求放在 **MySQL 更自然**：建索引、SQL 分页/模糊查询即可。
- **简单 key-value 访问**（如：根据 guildId 取公会、根据 playerId 取所属公会）：
  - 用 Redis 做缓存或直接存，性能更好；MySQL 做持久化与兜底。

### 4. 典型分层

```
┌─────────────────────────────────────────────────────────────┐
│  业务层（内存 / 应用）                                         │
│  - 热点公会 entity、在线成员、广播等                            │
└─────────────────────────────────────────────────────────────┘
         │ 读/写
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Redis：缓存 + 会话 + 排行榜 + 聊天                            │
│  - 玩家公会关系 player_guild:{id}                             │
│  - 公会信息缓存 guild:{id}（可选）                              │
│  - 排行榜、聊天记录、申请列表等                                 │
└─────────────────────────────────────────────────────────────┘
         │ 缓存未命中 / 写回
         ▼
┌─────────────────────────────────────────────────────────────┐
│  MySQL：持久化 + 结构化查询                                    │
│  - 公会主表 guild（含成员列表等）                               │
│  - 公会日志 guild_log                                        │
│  - 需要名字唯一、分页、搜索时查 MySQL（或通过缓存预热）           │
└─────────────────────────────────────────────────────────────┘
```

## 四、对当前实现的评价与建议

### 做得好的地方

1. **公会主数据落 MySQL**：持久化、可恢复、可做名字唯一约束和后续扩展查询，方向正确。
2. **公会日志用 MySQL、按 guild_id 查**：追加型、按维度查询，适合关系型存储。
3. **玩家-公会关系放 Redis**：按 playerId 访问，写多读多都适合，和「会话/状态」语义一致。
4. **排行榜、聊天用 Redis**：符合常见做法。

### 可以改进的点

1. **Manager 全量 JSON 进 Redis（GuildManager key）**  
   - 问题：Manager 里包含大量与 MySQL 重复的 guild 数据，容易不一致；且单 key 过大时，网络与序列化成本高。  
   - 建议：Redis 只存「轻量索引」或不再存 Manager 快照；**启动时以 MySQL 为唯一数据源**加载公会列表和公会数据，Redis 只做玩家侧和排行榜/聊天等。

2. **公会名唯一校验**  
   - 当前：只依赖 MySQL `WHERE guild_name = ?`，正确。  
   - 若后续做「搜索公会」、分页，也建议以 MySQL 为主（建索引、分页查询），必要时再配合 Redis 缓存热门结果。

3. **公会列表/分页（OnGetGuildListByPage、OnSearchGuildByName）**  
   - 当前：遍历内存 `guildList`，公会多了会有性能和单机内存上限问题。  
   - 建议：  
     - **列表/分页/按条件筛选**：走 MySQL（如 `guild` 表加索引，分页查）；  
     - 热点公会可缓存到 Redis（如 `guild:info:{id}`），而不是把所有公会都放在单机内存里。

4. **写回策略**  
   - 当前：30 分钟分批写回 MySQL，存在 30 分钟内宕机丢数据的风险。  
   - 建议：关键变更（创建/解散/踢人/任命等）**关键路径同步写 MySQL**，其余可异步/批量写回；或至少对「结构性变更」做即时落库，再配合 Redis 缓存。

5. **成员列表存在 MySQL 的方式**  
   - 当前：`MemberData` 以 map 形式放在 `guild` 表（可能是 JSON/Blob），不利于按成员维度的查询（如「某玩家加入了哪些公会」的历史）。  
   - 若未来需要「成员维度查询」或更细的权限/审计，可考虑拆出 **guild_member 表**（guild_id, player_id, position, join_time…），公会主表只保留核心元信息；当前若仅按公会维度访问，维持现状也可接受。

## 五、结论与选型建议

| 问题 | 建议 |
|------|------|
| 工会数据到底用 MySQL 还是 Redis？ | **不要二选一**：MySQL 做**持久化与结构化查询**，Redis 做**缓存、玩家维度状态、排行榜、聊天**。 |
| 当前用 MySQL 存公会主数据是否妥当？ | **妥当**。公会主数据适合放在 MySQL，便于持久化、唯一约束、分页与搜索。 |
| 是否要改成「全 Redis」？ | **不建议**。全 Redis 在持久化、备份、复杂查询、运维上成本更高，业界较少把公会主数据只放 Redis。 |
| 是否要改成「全 MySQL」？ | **不推荐**。玩家-公会关系、排行榜、聊天等高并发、按 key 访问的场景，用 Redis 更合适；若全用 MySQL，容易成为瓶颈。 |

**推荐方向（与业界通用做法一致）：**

- **MySQL**：公会主表（含或拆出成员）、公会日志；所有「需要持久化、唯一约束、分页、搜索」的数据。
- **Redis**：玩家-公会关系、公会信息缓存（可选）、排行榜、聊天、申请列表等；必要时启动时从 MySQL 加载公会列表，而不依赖 Redis 上的 Manager 全量快照。
- **内存**：仅保留**热点公会**或**当前服关心的公会**的 entity，而不是无上限全量加载；列表/搜索走 MySQL（+ 缓存），避免单机内存和遍历瓶颈。

这样既保留「结构化查询」的便利性（MySQL），又保留高并发与低延迟（Redis + 内存缓存），职责清晰，也便于后续扩展（如多服、分库分表）。
