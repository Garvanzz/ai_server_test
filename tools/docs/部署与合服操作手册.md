## 1. 系统架构概览

每一台"区服"由以下进程组成：

```
┌─────────────────────────────────────────────────────┐
│                   客户端 (Client)                    │
└───────────┬─────────────────────────┬───────────────┘
            │ 登录/选服               │ 游戏长连接
            ▼                         ▼
┌─────────────────┐        ┌──────────────────────────┐
│  login_server   │        │       main_server         │
│  (HTTP :9007)   │        │  HTTP :9505               │
│  认证 / 服务器列表 │        │  WS   :8081  (WebSocket)  │
└────────┬────────┘        │  TCP  :8082  (Gate)       │
         │                 └────────────┬─────────────┘
         │                              │
         ▼                              ▼
┌──────────────────────────────────────────────────────┐
│                     MySQL (wenjiacheng)               │
│                     Redis (共享/独立)                 │
└──────────────────────────────────────────────────────┘
         ↑
┌────────┴────────┐
│   gm_server      │  ← 后台管理（HTTP :9004）
│   Admin 面板      │  ← Vue 前端（由 Nginx/Node 托管）
└─────────────────┘
```

**核心设计原则：**

| 概念 | 说明 |
|------|------|
| `entry_server_id` | 客户端选服看到的 ID，**永远不变** |
| `logic_server_id` | 实际业务数据所在的服，合服后指向目标服 |
| `origin_server_id` | 玩家最初创建角色的服，审计用，不可修改 |

## 2. 环境依赖

在每台服务器上完成以下安装：

### 2.1 基础环境

| 依赖 | 推荐版本 | 用途 |
|------|----------|------|
| Linux (CentOS 7+ / Ubuntu 20+) | — | 操作系统 |
| Go | 1.21+ | 编译所有 Go 服务 |
| MySQL | 5.7+ / 8.0 | 主数据库 |
| Redis | 6.0+ | 会话 / 游戏数据缓存 |
| Nginx | 1.20+ | 反向代理 / 前端托管 |
| Node.js + pnpm | 18+ / 8+ | 前端构建（仅 Admin 服务器) |

### 2.2 网络规划示例（两台区服 + 共用基础设施）

```
192.168.1.10  MySQL（共享）
192.168.1.11  Redis（共享）
192.168.1.20  gm_server + Admin 面板 + login_server
192.168.1.21  Server 1 (main_server)  ID=1
192.168.1.22  Server 2 (main_server)  ID=2
```

> **建议**：MySQL、Redis 使用共享实例，可降低合服时的数据迁移复杂度。

## 3. 数据库初始化

> **只需执行一次**，所有区服共用同一套表结构。

### 3.1 创建数据库

```sql
CREATE DATABASE wenjiacheng CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'game'@'%' IDENTIFIED BY 'your_password';
GRANT ALL PRIVILEGES ON wenjiacheng.* TO 'game'@'%';
FLUSH PRIVILEGES;
```

### 3.2 初始化完整 Schema

```bash
mysql -u game -p wenjiacheng < tools/scripts/mysql/schema_full.sql
```

此脚本将创建所有必要的表，包括：

| 表名 | 用途 |
|------|------|
| `account` | 全局账号信息 |
| `account_role` | 账号与区服角色映射（支持多服多角色） |
| `game_server` | 区服路由配置（entry_server_id 与 logic_server_id 映射） |
| `server_group` | 区服分组（用于前端显示) |
| `server_process` | 进程管理（启动/停止/编译配置） |
| `merge_plan` | 合服工单 |
| `merge_server_map` | 合服进度追踪 |
| `merge_conflict_log` | 合服冲突记录 |
| `guild` 等业务表 | 行会、邮件、好友等业务数据 |
| `pay_order` | 支付订单 |
| `hot_update` | 热更版本管理 |

### 3.3 后续增量迁移（升级已有环境时使用）

```bash
# 按编号顺序依次执行
mysql -u game -p wenjiacheng < tools/scripts/mysql/migrations/001_game_server_add_main_server_http_url.sql
mysql -u game -p wenjiacheng < tools/scripts/mysql/migrations/002_account_role_refactor.sql
mysql -u game -p wenjiacheng < tools/scripts/mysql/migrations/003_game_server_runtime_fields.sql

mysql -u game -p wenjiacheng < tools/scripts/mysql/migrations/006_game_server_drop_process_fields.sql
```

## 5. 从零部署：第一台区服（服务器 1）

> 服务器 1 的 `logic_server_id = 1`，`entry_server_id = 1`。

### 步骤 1：编译所有服务

```bash
cd /home/game/ai_server_test

# 编译 login_server
go build -o bin/login_server ./login_server/

# 编译 main_server
go build -o bin/main_server ./main_server/run/

# 编译 gm_server（如果部署在此机器上）
go build -o bin/gm_server ./gm_server/
```

### 步骤 2：创建目录结构

```bash
mkdir -p /home/game/server1/{login_server,main_server,gm_server,log}
```

### 步骤 3：部署 login_server

```bash
# 复制二进制和配置
cp bin/login_server /home/game/server1/login_server/
cp -r login_server/conf /home/game/server1/login_server/

# 修改配置（参见 4.1 节）
vim /home/game/server1/login_server/conf/loginserver.json

# 启动
cd /home/game/server1/login_server
nohup ./login_server > log/login.log 2>&1 &

# 验证
curl http://127.0.0.1:9007/health
```

### 步骤 4：部署 main_server（服务器 1）

```bash
# 复制二进制、配置和游戏配置
cp bin/main_server /home/game/server1/main_server/
cp main_server/run/env.toml /home/game/server1/main_server/
cp -r core/config/json /home/game/server1/main_server/config_json

# 编辑 env.toml，将 ID 设为 1
vim /home/game/server1/main_server/env.toml
# 确认：ID = 1

# 启动
cd /home/game/server1/main_server
nohup ./main_server > log/main.log 2>&1 &

# 验证 HTTP 端口
curl http://127.0.0.1:9505/health
```

### 步骤 5：在数据库中注册区服 1

**方式 A：通过 Admin 面板（推荐）**

1. 打开 Admin 面板 → 服务器管理 → 大厅服列表
2. 点击「新增大厅服」，填写：

| 字段 | 值 |
|------|----|
| 服务器ID | 1 |
| 服务器名称 | 一区 |
| 区服组 | 选择已有分组 |
| IP | 192.168.1.21 |
| 端口 | 9505 |
| 大厅服HTTP | http://192.168.1.21:9505 |
| 服务器状态 | 正常 (0) |
| 开服时间 | 按实际填写 |

**方式 B：直接写库**

```sql
INSERT INTO game_server (id, channel, group_id, logic_server_id, ip, port, main_server_http_url, server_state, server_name, open_server_time)
VALUES (1, 0, 1, 1, '192.168.1.21', 9505, 'http://192.168.1.21:9505', 0, '一区', '2026-04-01 00:00:00');
```

### 步骤 6：在数据库中注册进程记录（可选，用于 GM 远程管理）

**通过 Admin 面板 → 进程管理 → 新增进程**，填写：

| 字段 | 值 |
|------|----|
| 进程名称 | 一区-main_server |
| 进程类型 | 大厅服 (2) |
| 关联服ID | 1 |
| 管理方式 | local_command |
| 可执行文件名 | main_server |
| 启动命令 | /home/game/server1/main_server/main_server |

## 10. 合服操作

> **前提**：服务器 2（二区）已运行一段时间，现需将二区数据合入一区。合并后：
> - 玩家仍可通过"二区"入口登录，但实际数据在"一区"服务器上处理。
> - `entry_server_id=2` 保留，`logic_server_id=2 → 1`。

### 10.1 合服前准备（重要，不可省略）

#### 第一步：安排维护窗口，通知玩家

通过 Admin 面板 → 通知管理，发送停服公告。

#### 第二步：将两台区服设为维护状态

Admin 面板 → 大厅服列表 → 选中服务器 1 和服务器 2 → 批量维护

或执行：

```sql
UPDATE game_server SET server_state = 3 WHERE id IN (1, 2);
-- server_state=3 表示维护中，登录服不再返回这两台服
```

#### 第三步：等待在线玩家全部下线

```bash
# 在 main_server 1 的管理接口上查询在线人数（具体接口参照项目实现）
curl http://192.168.1.21:9505/gm/online-count
# 直到返回 0
```

#### 第四步：停止两台 main_server

```bash
# Admin 面板 → 进程管理 → 找到服务器 1 和服务器 2 的进程 → 停止
# 或手动 kill
kill $(pgrep -f "server2/main_server")
# 服务器 1 保持运行，因为合服完成后数据在服务器 1 上
```

#### 第五步：完整备份数据库和 Redis

```bash
# MySQL 全量备份
mysqldump -u game -p wenjiacheng > backup_before_merge_$(date +%Y%m%d_%H%M%S).sql

# Redis 备份（如使用独立 Redis）
redis-cli -h 192.168.1.11 BGSAVE
cp /var/lib/redis/dump.rdb backup_redis_$(date +%Y%m%d_%H%M%S).rdb
```

**⚠️ 备份完成前，绝对不要执行任何合服步骤。**

---

### 10.2 执行合服

#### 第一步：打开 Admin 面板 → 合服管理

#### 第二步：预检查（Precheck）

填写：
- **目标服（Target）**：1（一区，数据保留方）
- **来源服（Source）**：2（二区，数据迁入方）

点击「预检查」，系统会返回：
- 行会名称冲突列表（若有，系统会自动在合服时重命名为 `{原名}_S2`）
- 各服数据量统计
- 操作风险提示

仔细阅读冲突列表，确认可接受后继续。

#### 第三步：创建合服工单

点击「创建工单」，填写：
- **工单名称**：`2026-04-01 二区合入一区`
- **目标服**：1
- **来源服**：2
- **备注**：按需填写

工单创建后状态为「待执行」。

#### 第四步：执行合服

在工单列表中找到刚创建的工单，点击「执行」。

系统将自动完成以下操作（按顺序）：

```
合服执行流程：
1. 重命名有冲突的行会名（guild 表）
   → 来源服 guild 中与目标服同名的，追加 _S2 后缀

2. 迁移业务数据（批量 UPDATE server_id: 2 → 1）
   - guild, guild_apply, guild_log
   - sys_mail_info, admin_mail, player_mail_info
   - friend_apply, friend_block
   - pay_order, pay_cache_order

3. 更新角色路由（account_role 表）
   UPDATE account_role SET logic_server_id = 1 WHERE logic_server_id = 2

4. 更新区服路由（game_server 表）
   UPDATE game_server SET logic_server_id = 1, merge_state = 2 WHERE id = 2

5. 更新工单状态为「成功」
```

执行完成后，工单状态变为「已完成」。

#### 第五步：验证合服结果

```sql
-- 验证路由更新
SELECT id, server_name, logic_server_id, merge_state FROM game_server WHERE id IN (1, 2);
-- 期望：id=2 的 logic_server_id=1, merge_state=2

-- 验证角色路由
SELECT entry_server_id, logic_server_id, COUNT(*) as cnt
FROM account_role
WHERE entry_server_id IN (1, 2)
GROUP BY entry_server_id, logic_server_id;
-- 期望：entry_server_id=2 的所有记录 logic_server_id 均为 1

-- 验证业务数据
SELECT server_id, COUNT(*) FROM guild GROUP BY server_id;
-- 期望：没有 server_id=2 的行会记录（全部已变为 1）
```

#### 第六步：处理 Redis（如使用独立 Redis）

如果服务器 1 和服务器 2 使用**独立 Redis**（不同实例），需要额外迁移 Redis 数据：

Admin 面板 → 合服管理 → Redis 检查

或手动导出迁移脚本：

```
POST /gm/merge/redis-script
{
  "targetServerId": 1,
  "sourceServerIds": [2],
  "redisMode": "independent"
}
```

系统会生成如下类型的迁移脚本（需在 Redis 层面执行）：

```bash
# 迁移玩家数据 Key
# 格式：RENAME Player:<serverId>:<playerId> Player:<targetServerId>:<playerId>
# 迁移行会数据
# RENAME GuildManager:2:* → GuildManager:1:*
# 迁移排行榜数据
# RENAME rank_2_* → rank_1_*
```

**⚠️ 如使用共享 Redis（推荐），则跳过此步骤。**

---

### 10.3 重启服务（合服后）

#### 第一步：启动服务器 1（一区）main_server

```bash
# 只需重启服务器 1，服务器 2 不再需要独立运行
cd /home/game/server1/main_server
nohup ./main_server > log/main.log 2>&1 &
```

#### 第二步：恢复区服状态

```sql
-- 将两台区服恢复为正常状态
UPDATE game_server SET server_state = 0 WHERE id IN (1, 2);
```

或 Admin 面板 → 大厅服列表 → 选中 → 批量取消维护。

#### 第三步：验证玩家登录

- 原"一区"玩家：正常登录，数据无变化
- 原"二区"玩家：选择"二区"入口登录，实际连接到"一区"服务器，数据合并成功

## 11. 合服回滚

> **⚠️ 警告**：GM 提供的回滚功能**只回滚路由层（game_server 表的 logic_server_id）**，**不回滚已迁移的业务数据（MySQL 和 Redis）**。若需要完整回滚，必须从备份恢复。

### 11.1 路由层回滚（快速恢复访问）

Admin 面板 → 合服管理 → 找到工单 → 点击「回滚」

或调用接口：

```
POST /gm/merge/rollback
{"planId": <工单ID>}
```

此操作会还原：

```sql
UPDATE game_server SET logic_server_id = id, merge_state = 0 WHERE id = 2;
-- 即 logic_server_id 恢复为原始值（等于自身 id）
```

### 11.2 完整数据回滚（从备份恢复）

```bash
# 1. 停止所有 main_server
kill $(pgrep -f "main_server")

# 2. 恢复 MySQL
mysql -u game -p wenjiacheng < backup_before_merge_20260401_000000.sql

# 3. 恢复 Redis（如独立 Redis）
redis-cli -h 192.168.1.11 SHUTDOWN NOSAVE
cp backup_redis_20260401_000000.rdb /var/lib/redis/dump.rdb
systemctl start redis

# 4. 重启服务器 1 和服务器 2 的 main_server
cd /home/game/server1/main_server && nohup ./main_server > log/main.log 2>&1 &
cd /home/game/server2/main_server && nohup ./main_server > log/main.log 2>&1 &

# 5. 恢复区服状态
# UPDATE game_server SET server_state = 0 WHERE id IN (1, 2);
```

## 12. 常见问题

### Q1：合服后二区玩家登录报"找不到玩家数据"

**原因**：`account_role` 路由更新成功，但 Redis 数据未迁移（独立 Redis 场景）。

**解决**：
1. 用 Admin 面板生成 Redis 迁移脚本并执行
2. 或强制玩家重新走"冷启动"流程（从 MySQL 加载数据到 Redis）

---

### Q2：合服执行一半报错（部分数据已迁移）

**原因**：合服执行不在事务中，部分表已更新，部分未更新。

**解决**：
1. **不要**再次点击「执行」
2. 查看工单状态，找到失败的来源服 ID
3. 查看 `merge_conflict_log` 表了解错误详情
4. 手动修复或从备份恢复
5. 如手动修复：对未迁移的表重新执行 `UPDATE ... WHERE server_id = <sourceId>`

---

### Q3：合服后出现行会名冲突

**原因**：precheck 时未发现或手动跳过了冲突检查。

**解决**：合服时系统自动将冲突行会重命名为 `{原名}_S2`，可在 Admin 面板 → 合服冲突记录中查看并通知玩家手动改名。

---

### Q4：同一台物理机部署两台 main_server

部署两台 main_server 在同一机器时，需确保：

```toml
# server1/env.toml
ID = 1
HttpUrl = "0.0.0.0:9505"
RemotePort = 1001

# server2/env.toml
ID = 2
HttpUrl = "0.0.0.0:9506"   # ← 端口不同
RemotePort = 1002            # ← 端口不同
```

同时在 `game_server` 表中对应修改 `main_server_http_url` 和 `port`。

---

### Q5：如何一次性合并多台区服到同一目标

在「创建工单」时，`sourceServerIds` 可以填写多个：

```json
{
  "name": "三区四区合并至一区",
  "targetServerId": 1,
  "sourceServerIds": [3, 4],
  "remark": ""
}
```

系统会依次对每个来源服执行完整的数据迁移流程。
